## Overview

AVA uses a sophisticated role-based access control (RBAC) system to ensure users have appropriate access to data governance features and data assets. This guide covers user management, roles, permissions, and best practices for access control.

<Info>
  AVA follows the principle of least privilege - users are granted only the minimum permissions necessary to perform their job functions.
</Info>

## User Management

### Create Users

<Tabs>
  <Tab title="Via Dashboard">
    <Steps>
      <Step title="Navigate to Users">
        Go to **Settings** > **Users** > **Add User**
      </Step>
      <Step title="Enter Details">
        - Email address
        - First and last name
        - Department
        - Manager
      </Step>
      <Step title="Assign Roles">
        Select one or more roles for the user
      </Step>
      <Step title="Set Permissions">
        Configure additional permissions or restrictions
      </Step>
      <Step title="Send Invitation">
        User receives email with setup instructions
      </Step>
    </Steps>
  </Tab>

  <Tab title="Via CLI">
    ```bash
    # Create single user
    ava users create \
      --email john.doe@company.com \
      --name "John Doe" \
      --role data-governance-officer \
      --department "Data Management" \
      --send-invite

    # Bulk import users
    ava users import --file users.csv

    # CSV format:
    # email,name,role,department
    # jane@company.com,Jane Smith,data-steward,Finance
    # bob@company.com,Bob Johnson,compliance-officer,Legal
    ```
  </Tab>

  <Tab title="Via API">
    ```bash
    curl -X POST https://ava.your-domain.com/api/v1/users \
      -H "Authorization: Bearer YOUR_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{
        "email": "john.doe@company.com",
        "name": "John Doe",
        "roles": ["data-governance-officer"],
        "department": "Data Management",
        "send_invite": true
      }'
    ```
  </Tab>
</Tabs>

### Manage User Roles

```bash
# Assign role to user
ava users assign-role \
  --email john.doe@company.com \
  --role compliance-officer

# Remove role from user
ava users remove-role \
  --email john.doe@company.com \
  --role data-steward

# Update user's primary role
ava users update john.doe@company.com \
  --primary-role administrator
```

### Deactivate Users

```bash
# Deactivate user (preserves data)
ava users deactivate john.doe@company.com

# Reactivate user
ava users activate john.doe@company.com

# Delete user (removes all data)
ava users delete john.doe@company.com --confirm
```

## Custom Roles

Create custom roles tailored to your organization:

<Steps>
  <Step title="Define Role">
    ```bash
    ava roles create \
      --name "Marketing Data Analyst" \
      --description "Access to marketing data only"
    ```
  </Step>

  <Step title="Assign Permissions">
    ```bash
    ava roles add-permission "Marketing Data Analyst" \
      --permission "data:view_marketing_sources" \
      --permission "data:query_marketing_data" \
      --permission "reports:create" \
      --permission "reports:view_own"
    ```
  </Step>

  <Step title="Set Resource Constraints">
    ```bash
    ava roles set-constraints "Marketing Data Analyst" \
      --data-sources "salesforce_marketing,google_analytics" \
      --tags "marketing,campaign" \
      --max-query-results 10000
    ```
  </Step>

  <Step title="Assign to Users">
    ```bash
    ava users assign-role \
      --email marketing-user@company.com \
      --role "Marketing Data Analyst"
    ```
  </Step>
</Steps>

## Data-Level Security

### Row-Level Security

Restrict data access based on user attributes:

```bash
# Create row-level security policy
ava security create-rls-policy \
  --name "Regional Data Access" \
  --data-source salesforce \
  --table Account \
  --filter "Region = {{user.region}}" \
  --roles "data-analyst,data-steward"

# Example: Users only see accounts in their region
```

### Column-Level Security

Mask or hide sensitive columns:

```bash
# Hide sensitive columns
ava security hide-columns \
  --data-source salesforce \
  --table Contact \
  --columns "SSN,CreditCardNumber" \
  --except-roles "administrator,compliance-officer"

# Mask columns (show partial data)
ava security mask-columns \
  --data-source salesforce \
  --table Contact \
  --column "Email" \
  --mask-type "email" \
  --roles "data-analyst"
```

### Attribute-Based Access Control (ABAC)

Define access based on attributes:

```python
# Python SDK example
from ava import Client

client = Client(api_key="YOUR_API_KEY")

# Create ABAC policy
policy = client.security.create_abac_policy(
    name="Healthcare Data Access",
    rules=[
        {
            "resource": {"classification": "PHI"},
            "subject": {"department": "Healthcare"},
            "action": ["read", "query"],
            "conditions": [
                "user.certified == true",
                "user.training_date > date('2024-01-01')"
            ]
        }
    ]
)
```

## Access Requests

Enable self-service access requests:

### Configure Access Request Workflow

```bash
ava workflows configure access-request \
  --approvers "data-governance-team@company.com" \
  --auto-approve-duration 7d \
  --require-justification \
  --notify-owner
```

### Request Access (User Perspective)

<Steps>
  <Step title="Find Resource">
    User browses data catalog and finds needed resource
  </Step>
  <Step title="Request Access">
    Clicks "Request Access" and fills out form:
    - Justification
    - Duration needed
    - Access level required
  </Step>
  <Step title="Approval Process">
    Request routes to appropriate approver(s)
  </Step>
  <Step title="Access Granted">
    User receives notification when approved
  </Step>
</Steps>

### Approve Access Requests

```bash
# List pending requests
ava access-requests list --status pending

# Approve request
ava access-requests approve req_12345 \
  --duration 30d \
  --comment "Approved for project X"

# Deny request
ava access-requests deny req_12345 \
  --reason "Insufficient justification"
```

## Groups and Teams

Organize users into groups for easier management:

### Create Groups

```bash
# Create group
ava groups create \
  --name "Data Governance Team" \
  --description "Primary data governance team" \
  --email "data-governance@company.com"

# Add members
ava groups add-members "Data Governance Team" \
  --users "john@company.com,jane@company.com,bob@company.com"

# Assign role to entire group
ava groups assign-role "Data Governance Team" \
  --role data-governance-officer
```

### Group-Based Permissions

```bash
# Grant group access to data source
ava access grant \
  --group "Data Governance Team" \
  --resource "snowflake_prod" \
  --permission "read,query"

# Set as data owner group
ava data-sources set-owner snowflake_prod \
  --group "Data Governance Team"
```

## Service Accounts

Create service accounts for automation:

```bash
# Create service account
ava service-accounts create \
  --name "ETL Pipeline" \
  --description "Automated ETL process" \
  --email "etl-service@company.com"

# Generate API key
ava service-accounts create-key "ETL Pipeline" \
  --expires-in 90d \
  --scopes "data:read,data:query"

# Assign minimal permissions
ava service-accounts assign-permissions "ETL Pipeline" \
  --permissions "data:query_source:snowflake_prod" \
  --permissions "data:read_catalog"
```

<Warning>
  Service account credentials should be stored securely and rotated regularly.
</Warning>

## Permission Auditing

### Monitor User Activity

```bash
# View user's recent activity
ava audit user-activity john.doe@company.com \
  --days 30

# See what data a user has accessed
ava audit data-access \
  --user john.doe@company.com \
  --include-queries

# Check permission changes
ava audit permission-changes \
  --days 7
```

### Access Reviews

Periodic review of user permissions:

```bash
# Generate access review report
ava access-review generate \
  --department "Marketing" \
  --output access-review-2024-10.pdf

# Export user permissions
ava users export-permissions \
  --format csv \
  --output user-permissions.csv
```

## Troubleshooting

<AccordionGroup>
  <Accordion title="User cannot access resource">
    **Check:**
    1. User has appropriate role
    2. Resource is included in role's scope
    3. No conflicting deny policies
    ```bash
    ava access check \
      --user john@company.com \
      --resource salesforce_prod \
      --action read
    ```
  </Accordion>

  <Accordion title="Permission changes not taking effect">
    **Solutions:**
    1. User may need to log out and back in
    2. Clear session cache
    3. Check for conflicting policies
    ```bash
    ava users refresh-permissions john@company.com
    ```
  </Accordion>

  <Accordion title="Cannot create custom role">
    **Verify:**
    1. You have `manage_roles` permission
    2. Role name is unique
    3. Permissions are valid
    ```bash
    ava roles validate --config role-definition.yaml
    ```
  </Accordion>
</AccordionGroup>

## Security Considerations

<Info>
  **Security Tips:**
  - Enable MFA for all users, especially admins
  - Regularly review and revoke unused API keys
  - Monitor for privilege escalation attempts
  - Use time-limited access grants when possible
  - Maintain audit logs for compliance
</Info>
