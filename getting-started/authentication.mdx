## Authentication Methods

### Single Sign-On (SSO)

Configure SAML 2.0 SSO with your identity provider:

<Tabs>
  <Tab title="Okta">
    <Steps>
      <Step title="Create SAML App in Okta">
        1. Log in to your Okta Admin Console
        2. Navigate to **Applications** > **Applications**
        3. Click **Create App Integration**
        4. Select **SAML 2.0** and click **Next**
      </Step>

      <Step title="Configure SAML Settings">
        **App Settings:**
        - **App name**: AVA by DataRM
        - **Single sign on URL**: `https://ava.your-domain.com/auth/saml/callback`
        - **Audience URI (SP Entity ID)**: `https://ava.your-domain.com`
        - **Name ID format**: EmailAddress
        - **Application username**: Email
      </Step>

      <Step title="Configure in AVA">
        ```bash
        ava auth configure saml \
          --provider okta \
          --entity-id "your-okta-entity-id" \
          --sso-url "https://your-domain.okta.com/app/xxx/sso/saml" \
          --certificate-path /path/to/okta-cert.pem
        ```
      </Step>

      <Step title="Test Connection">
        ```bash
        ava auth test --provider okta
        ```
      </Step>
    </Steps>

    <Check>
      Users can now sign in to AVA using their Okta credentials!
    </Check>
  </Tab>

  <Tab title="Azure AD">
    <Steps>
      <Step title="Register App in Azure AD">
        1. Navigate to **Azure Active Directory** > **Enterprise applications**
        2. Click **New application** > **Create your own application**
        3. Enter name: "AVA by DataRM"
        4. Select **Integrate any other application you don't find in the gallery**
      </Step>

      <Step title="Configure Single Sign-On">
        1. Select **Single sign-on** > **SAML**
        2. Click **Edit** on Basic SAML Configuration

        **Settings:**
        - **Identifier (Entity ID)**: `https://ava.your-domain.com`
        - **Reply URL**: `https://ava.your-domain.com/auth/saml/callback`
        - **Sign on URL**: `https://ava.your-domain.com/login`
      </Step>

      <Step title="Download Certificate">
        Download the **Certificate (Base64)** from the SAML Signing Certificate section
      </Step>

      <Step title="Configure in AVA">
        ```bash
        ava auth configure saml \
          --provider azure \
          --entity-id "https://sts.windows.net/your-tenant-id/" \
          --sso-url "https://login.microsoftonline.com/your-tenant-id/saml2" \
          --certificate-path /path/to/azure-cert.pem
        ```
      </Step>
    </Steps>
  </Tab>

  <Tab title="Google Workspace">
    <Steps>
      <Step title="Create SAML App">
        1. Go to **Google Admin Console** > **Apps** > **Web and mobile apps**
        2. Click **Add app** > **Add custom SAML app**
        3. Enter app name: "AVA by DataRM"
      </Step>

      <Step title="Download Metadata">
        Download the IdP metadata or note:
        - SSO URL
        - Entity ID
        - Certificate
      </Step>

      <Step title="Configure Service Provider Details">
        - **ACS URL**: `https://ava.your-domain.com/auth/saml/callback`
        - **Entity ID**: `https://ava.your-domain.com`
        - **Start URL**: `https://ava.your-domain.com/login`
        - **Name ID format**: EMAIL
      </Step>

      <Step title="Configure in AVA">
        ```bash
        ava auth configure saml \
          --provider google \
          --entity-id "your-google-entity-id" \
          --sso-url "https://accounts.google.com/o/saml2/idp?idpid=xxx" \
          --certificate-path /path/to/google-cert.pem
        ```
      </Step>
    </Steps>
  </Tab>
</Tabs>

### OAuth 2.0 / OpenID Connect

Configure OAuth authentication with popular providers:

<CodeGroup>
```bash Okta OAuth
ava auth configure oauth \
  --provider okta \
  --client-id "your-client-id" \
  --client-secret "your-client-secret" \
  --issuer "https://your-domain.okta.com" \
  --scopes "openid profile email groups"
```

```bash Auth0
ava auth configure oauth \
  --provider auth0 \
  --client-id "your-client-id" \
  --client-secret "your-client-secret" \
  --issuer "https://your-tenant.auth0.com" \
  --scopes "openid profile email"
```

```bash Generic OIDC
ava auth configure oauth \
  --provider generic \
  --client-id "your-client-id" \
  --client-secret "your-client-secret" \
  --issuer "https://your-idp.com" \
  --authorization-endpoint "https://your-idp.com/oauth/authorize" \
  --token-endpoint "https://your-idp.com/oauth/token" \
  --userinfo-endpoint "https://your-idp.com/oauth/userinfo"
```
</CodeGroup>

### LDAP / Active Directory

Integrate with your on-premises directory services:

```bash
ava auth configure ldap \
  --host "ldap.company.com" \
  --port 636 \
  --use-ssl \
  --bind-dn "CN=AVA Service Account,OU=Service Accounts,DC=company,DC=com" \
  --bind-password "your-service-account-password" \
  --search-base "DC=company,DC=com" \
  --search-filter "(|(sAMAccountName={{username}})(mail={{username}}))" \
  --group-search-base "OU=Groups,DC=company,DC=com" \
  --group-search-filter "(member={{dn}})"
```

<Accordion title="Advanced LDAP Configuration">
  **Attribute Mapping:**
  ```bash
  ava auth ldap set-attributes \
    --email-attribute "mail" \
    --firstname-attribute "givenName" \
    --lastname-attribute "sn" \
    --username-attribute "sAMAccountName" \
    --group-attribute "memberOf"
  ```

  **Connection Pool Settings:**
  ```bash
  ava auth ldap configure-pool \
    --min-connections 5 \
    --max-connections 20 \
    --connection-timeout 30
  ```
</Accordion>

## API Authentication

### API Keys

Generate and manage API keys for programmatic access:

<Steps>
  <Step title="Create API Key">
    Via Dashboard:
    1. Navigate to **Settings** > **API Keys**
    2. Click **Generate New Key**
    3. Enter description: "Production Service Account"
    4. Select scope and expiration
    5. Copy the generated key (shown only once!)

    Via CLI:
    ```bash
    ava api-key create \
      --name "Production Service" \
      --scopes "read:data,write:policies" \
      --expires-in 365d
    ```
  </Step>

  <Step title="Use API Key">
    ```bash
    # Set as environment variable
    export AVA_API_KEY="your-api-key-here"

    # Or pass in requests
    curl -X GET https://ava.your-domain.com/api/v1/data-sources \
      -H "Authorization: Bearer your-api-key-here"
    ```
  </Step>
</Steps>

<Warning>
  **Security Best Practices:**
  - Never commit API keys to version control
  - Use environment variables or secret managers
  - Rotate keys regularly (every 90 days recommended)
  - Use minimum required scopes
  - Set expiration dates
</Warning>

### Service Accounts

Create dedicated service accounts for automated workflows:

```bash
# Create service account
ava users create-service-account \
  --name "Data Pipeline Service" \
  --email "data-pipeline@company.com" \
  --role "service-account"

# Generate credentials
ava users generate-credentials "data-pipeline@company.com" \
  --output /secure/path/credentials.json
```

<Info>
  Service accounts are designed for machine-to-machine authentication and don't require interactive login.
</Info>

## Multi-Factor Authentication (MFA)

Enhance security by requiring MFA for all users:

### Enable MFA

```bash
# Enable MFA globally
ava auth mfa enable --enforce

# Enable for specific roles
ava auth mfa enable --roles admin,data-governance

# Enable with grace period
ava auth mfa enable --enforce --grace-period 7d
```

### Configure MFA for Users

<Tabs>
  <Tab title="User Self-Service">
    Users can enable MFA from their profile:
    1. Navigate to **Profile** > **Security**
    2. Click **Enable Two-Factor Authentication**
    3. Scan QR code with authenticator app
    4. Enter verification code to confirm
    5. Save backup codes securely
  </Tab>

  <Tab title="Admin Enforcement">
    Administrators can enforce MFA:
    ```bash
    # Require MFA for all admins
    ava auth mfa require --role admin

    # Send enrollment email to users
    ava auth mfa send-enrollment --all-users

    # Check MFA enrollment status
    ava auth mfa status
    ```
  </Tab>
</Tabs>

## Session Management

### Configure Session Settings

```bash
# Set session timeout
ava auth session set-timeout --duration 8h

# Enable "Remember Me" with 30-day duration
ava auth session enable-remember-me --duration 30d

# Configure concurrent session limit
ava auth session set-max-concurrent 3

# Enable session activity tracking
ava auth session enable-tracking
```

### Session Security

<AccordionGroup>
  <Accordion title="Force Logout on Password Change">
    ```bash
    ava auth session configure \
      --force-logout-on-password-change \
      --force-logout-on-role-change
    ```
  </Accordion>

  <Accordion title="Geographic Restrictions">
    ```bash
    # Allow login only from specific countries
    ava auth session set-geo-restrictions \
      --allowed-countries "US,CA,GB" \
      --block-vpn

    # Alert on unusual locations
    ava auth session enable-geo-alerts
    ```
  </Accordion>

  <Accordion title="Device Trust">
    ```bash
    # Require device registration
    ava auth device-trust enable

    # Set device re-authentication period
    ava auth device-trust set-reauth-period 30d
    ```
  </Accordion>
</AccordionGroup>

## Role-Based Access Control (RBAC)

Map authentication groups to AVA roles:

```bash
# Map SSO groups to AVA roles
ava auth role-mapping add \
  --group "datarm-admins" \
  --role "administrator"

ava auth role-mapping add \
  --group "datarm-data-governance" \
  --role "data-governance-officer"

ava auth role-mapping add \
  --group "datarm-compliance" \
  --role "compliance-officer"

# List all role mappings
ava auth role-mapping list
```

<Info>
  Learn more about roles and permissions in the [User Permissions](/getting-started/user-permissions) guide.
</Info>

## Testing Authentication

### Test SSO Configuration

```bash
# Test SAML authentication
ava auth test saml --verbose

# Test OAuth flow
ava auth test oauth --provider okta

# Test LDAP connection
ava auth test ldap --user "testuser@company.com"
```

### Validate User Login

```bash
# Simulate user login
ava auth validate-login \
  --username "user@company.com" \
  --show-groups \
  --show-roles
```

Expected output:
```json
{
  "status": "success",
  "user": {
    "email": "user@company.com",
    "name": "John Doe",
    "groups": ["datarm-users", "datarm-data-governance"],
    "roles": ["data-governance-officer"]
  },
  "authentication": {
    "method": "saml",
    "provider": "okta",
    "mfa_enabled": true
  }
}
```

## Troubleshooting

<AccordionGroup>
  <Accordion title="SAML authentication fails">
    **Common issues:**
    1. Clock skew between IdP and AVA
       ```bash
       # Sync system time
       sudo ntpdate -s time.nist.gov
       ```
    2. Invalid certificate
       ```bash
       # Verify certificate
       openssl x509 -in cert.pem -text -noout
       ```
    3. Incorrect URLs
       ```bash
       # Verify SAML configuration
       ava auth config show --provider saml
       ```
  </Accordion>

  <Accordion title="Users cannot see data after login">
    **Check role mappings:**
    ```bash
    # Show user's assigned roles
    ava users show user@company.com --include-roles

    # Verify role has appropriate permissions
    ava roles show data-governance-officer --include-permissions
    ```
  </Accordion>

  <Accordion title="LDAP connection timeouts">
    **Solutions:**
    1. Test network connectivity:
       ```bash
       telnet ldap.company.com 636
       ```
    2. Verify firewall rules allow LDAPS (port 636)
    3. Check service account permissions
    4. Increase connection timeout:
       ```bash
       ava auth ldap set-timeout 60
       ```
  </Accordion>

  <Accordion title="MFA enrollment issues">
    **Reset MFA for user:**
    ```bash
    # As admin, reset user's MFA
    ava auth mfa reset --user user@company.com

    # User will be prompted to re-enroll on next login
    ```
  </Accordion>
</AccordionGroup>
